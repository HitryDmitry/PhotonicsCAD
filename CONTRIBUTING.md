# Руководство для разработчиков (Contributing Guide)

Добро пожаловать в команду! Этот документ описывает, как мы работаем с Git и как вносить изменения в проект.

**Время чтения:** ~5 минут.

**Если нет опыта работы с git, необходимо пройти этот или похожий [туториал](https://githowto.com/ru).**

---

## 1. Общая концепция

Мы используем **GitHub Flow** — простой и гибкий подход, подходящий для нашей небольшой команды. Подробнее в [этой](https://habr.com/ru/articles/346066/) статье.

- **Одна основная ветка:** `main` — всегда стабильна и готова к использованию.
- **Всё остальное:** делаем в **feature-ветках** (от `main`).
- **Никаких долгоживущих веток** типа `develop`, `release` или `hotfix` у нас нет.

---

## 2. Ветки (Branches)

### Правила именования веток

Используйте префиксы для быстрой идентификации типа задачи:

| Префикс          | Пример                          | Когда использовать |
|------------------|--------------------------------|---------------------|
| `feature/`       | `feature/mmi-optimization`     | Новая фича или улучшение |
| `fix/`           | `fix/effective-index-bug`      | Исправление бага |
| `benchmark/`     | `benchmark/fdtd-performance`   | Эксперименты с производительностью |
| `wip/`           | `wip/new-waveguide-model`      | Черновик (work in progress) — НЕ для ревью |

### Создание новой ветки

Всегда создавайте ветку от актуальной `main`:

```bash
git checkout main
git pull origin main
git checkout -b feature/краткое-название-фичи
```

## 3. Коммиты (Commits)

Пишите осмысленные сообщения на русском или английском (единообразие важнее языка).

### Формат (нестрогий, но желательный):
```
feat: добавил расчет эффективного показателя для реберного волновода
fix: исправил ошибку сходимости в BPM-солвере при малом шаге сетки
perf: ускорил FFT для MMI-секций в 1.5 раза
docs: обновил пример в /examples/mzi.py
```
**ВАЖНО!** *Коммитьте чаще, но логически цельными кусками. Не коммитьте поломанный код (даже локально — лучше используйте stash).*

## 4. Pull Request (PR) — основной механизм интеграции
Любые изменения попадают в main только через Pull Request (или Merge Request в GitLab).

### Как открыть PR
1. Запушьте ветку в удалённый репозиторий:
```bash
git push origin feature/your-feature
```
2. Откройте PR на GitHub/GitLab в ветку `main`.
3. Заполните шаблон PR.

### Обязательные условия для влития (merge)
#### ✅ CI/CD прошёл успешно:
- Все юнит-тесты зелёные.
- Бенчмарки не показали деградацию производительности (если ваш код влияет на скорость).

#### ✅ Code Review:
- Как минимум один аппрув от другого члена команды.
- Кто открыл PR — тот и мержит (но не свой код без ревью!).

#### ✅ Нет конфликтов с `main`:

- Перед мержем обновите ветку:

```bash
git checkout feature/your-feature # Команда переключает фокус (вашу рабочую директорию) на ветку feature/your-feature, где вы разрабатываете свою задачу.

git pull origin main # Скачивает все изменения из ветки main и создает новый «коммит слияния» (merge commit) в вашей ветке.

# или
git pull origin main --rebase (или rebase): # Временно «откатывает» ваши локальные коммиты, применяет изменения из main (как если бы вы только что написали код поверх них), а затем возвращает ваши коммиты обратно.
```
- Рекомендуем rebase локально, но не переписывайте публичные ветки, если с ними кто-то работает.

### Кто делает ревью?
- Если PR касается численных методов / физики — обязательно участие хотя бы одного человека с профильным опытом.
- Если это чистый рефакторинг или документация — достаточно любого разработчика.

---
🚨 После мержа PR ветка **обязательно удаляется**. Если фича требует продолжения — создаётся новая ветка с новым именем.
---

## 5. Процесс принятия решений
- Голосование не требуется: технические решения принимает автор PR, но ревьювер может блокировать мерж, если есть серьёзные возражения.
- Спорные вопросы (например, выбор алгоритма) выносятся на общее обсуждение в чат команды или в комментарии к PR.

## 6. Релизы (Releases)
- Мы не используем ветки release. Вместо этого:
- Когда main накопил достаточное количество изменений — создаём тег:
```bash
git tag -a v1.2.3 -m "Описание релиза: улучшен MMI-солвер, исправлены баги с GDSII-экспортом"
git push origin v1.2.3
```
- Теги соответствуют [SemVer](https://semver.org/lang/ru/):
    - MAJOR — ломающие обратную совместимость изменения.
    - MINOR — новая функциональность (совместимая).
    - PATCH — исправления багов.

## 7. CI (Continuous Integration)
- Запускается на каждый PR и каждый коммит в main.
- Состоит из:
    - doctest — все модульные тесты.
    - Google Benchmark — проверка производительности критических функций (если добавлен бенчмарк).

Если CI упал — исправляйте до мержа. Исключение — если вы точно знаете, что тест устарел (тогда сначала исправьте тест в отдельном коммите).

## 8. Что делать, если срочно нужно исправить баг в уже выпущенной версии?
1. Создайте ветку fix/hotfix-description от тега старой версии.
2. Внесите исправление.
3. Откройте PR в main, но в описании укажите, что это критический баг-фикс.
4. После мержа в main — создайте новый патч-тег (например, v1.2.4).

**Важно**: Если вы правите старую версию, а main уже сильно ушёл вперёд — сделайте git cherry-pick нужных коммитов, а не мерж всей ветки.

## 9. Взаимодействие с примерами и документацией
- Если вы добавили новую фичу — обновите пример в папке /examples (например, новый скрипт для MZI или Y-разветвителя).
- Если изменили публичный API — обновите docstrings и, если есть, пользовательскую документацию в /docs.

## 10. Вопросы и контакты
Если что-то непонятно **— лучше спросить, чем гадать**.
- Задавайте вопросы в общем чате команды.
- Для технических обсуждений — создавайте Issue на GitHub (если используете) или обсуждайте прямо в PR.

## Памятка (шпаргалка)
```bash
# Начало работы над фичей
git checkout main && git pull
git checkout -b feature/your-feature

# В процессе (регулярно пушим)
git add .
git commit -m "feat: описание"
git push origin feature/your-feature

# Перед открытием PR — обновляем ветку
git fetch origin
git rebase origin/main
# (если конфликты — разрешаем локально)

# Создаём PR через веб-интерфейс
# Ждём CI и аппрув → мержим через интерфейс

# Релиз
git tag -a vX.Y.Z -m "текст"
git push origin vX.Y.Z
```
